## 🛠️ 关键技术决策与逻辑架构演进

### 1. 标签与书签的隔离记忆双轨制 (URL Memory)
- **痛点**：原本书签的标签（`bookmarkTags`）是绑定在 Chrome 原生书签 `ID` 上的。当用户删除书签时，该 `ID` 对应的标签关联也随之抹去。这导致下次收藏相同网址时，用户必须重新打标签。
- **决策**：引入以 URL 串作为主键的独立永久字典：`urlTagsMemory`。
  - 删除书签时，仅清理 `bookmarkTags` 中该 `ID` 的数据（维护后台整洁），但保留 `urlTagsMemory` 中该 URL 对应的标签历史。
  - 新增收藏或输入 URL 时，如果该 URL 在 `urlTagsMemory` 有过记录，则默认自动加载与继承。

### 2. 新建书签 URL 联动的智能覆盖/清空阻断
- **痛点**：如果在新建书签的弹出层，输入/修改 URL 会自动加载历史标签，可能会粗暴覆盖掉用户好不容易在弹窗里新打上去的标签。
- **决策**：通过建立 `lastLoadedUrl` 和 `lastLoadedTags` 监测机制：
  - 系统仅在当前标签区域与 `lastLoadedTags` 完全吻合（即用户在这期间**没有手动增加或删除过标签**）时，修改 URL 才会联动覆盖或清空。
  - 一旦监测到用户已经修改了任何一个标签，联动更新将自动挂起，只更新 `lastLoadedUrl`，最大程度保护用户的手动输入成果。
  - 当 URL 被清空或改回默认值（如 `https://`）时，同样会在未被篡改的前提下自动把标签还原为空，避免视觉残留。

### 3. 目录与全局视图数据流的一致性
- **痛点**：用户在全局视图（如“全部书签”）下设置了标签交集过滤，当其点击侧边栏的某具体“文件夹视图”后，如果只渲染该文件夹的子项而不隐藏顶部的过滤提示条，会导致用户视觉错乱。
- **决策**：设计了局部视图刷新与过滤器状态的高内聚解耦。一旦切换至任何具体的文件夹，顶部的标签过滤器会自动收起重置，确保单一视图的数据源和条件描述 100% 对应。
